```typescript
export async function createHousehold(data: unknown): Promise<
  { success: true; household: Household } | { error: string }
>;

// src/features/household/queries.ts
export async function getUserHouseholds(userId: string): Promise<
  Array<{ household: Household; role: "OWNER" | "MEMBER"; isDefault: boolean; joinedAt: Date }>
>;
```

**Test-isolation pattern (namespaced email prefix):**

```typescript
const testEmailPrefix = `integration-test-${Date.now()}-${crypto.randomUUID()}`;
// Every user created in the test gets `${testEmailPrefix}-userN@test.local`
// afterAll: delete all users (and cascaded rows) matching this prefix
```

This pattern is safe because:
- `User.email` is unique; collisions impossible
- Phase 1 set `Plant → User` to `SetNull` on delete, so other plants survive
- `HouseholdMember → User` cascade deletes the membership when the user goes away; household rows may remain (owner-free) — explicit cleanup for these too

**Assertion catalog (minimum 5 tests):**

| # | Test | Assertion |
|---|------|-----------|
| 1 | createHousehold happy path | Returns `{ success: true, household: {...} }`; household row exists in DB; OWNER HouseholdMember row exists with `userId`, `householdId`, `role: "OWNER"`, `rotationOrder: 0`, `isDefault: false` |
| 2 | createHousehold transaction atomicity | Mock/induce a failure in `householdMember.create` via extreme input → assert the Household row does NOT exist after rollback (use `db.household.findUnique({ where: { slug } })` → null) |
| 3 | createHousehold unique-slug enforcement | Create two households back-to-back; both succeed with DIFFERENT slugs (CUID-ish uniqueness verified at DB level) |
| 4 | getUserHouseholds with two memberships | Create User → create Household A (via registerUser — auto-default) → call `createHousehold` for Household B → `getUserHouseholds(userId)` returns 2 entries; A has `role: "OWNER", isDefault: true`, B has `role: "OWNER", isDefault: false`; sorted by joinedAt asc (A first) |
| 5 | getUserHouseholds returns empty for no-membership user | Create User (via `db.user.create` directly, NOT via registerUser — no auto-household); `getUserHouseholds(userId)` returns `[]` |

**Session mocking:**

`createHousehold` calls `auth()` internally. In an integration test we don't have a NextAuth session cookie. Two options:

Option A — `vi.mock("../auth", ...)` to return a fake session. The test still exercises real DB code path; only the session lookup is mocked. Trade-off: not "fully integration" but pragmatic.

Option B — use `next-auth/react` test helpers. Heavier setup; not worth it for Phase 2.

**Recommendation: Option A.** Explicitly document in the test file header that `auth()` is mocked, all other dependencies are real. This matches the goal of D-18 — catch transactional bugs, not session-plumbing bugs (which mocked unit tests cover).

**Database connectivity:**

Tests run against `DATABASE_URL` (dev DB). Do NOT spin up a containerised Postgres in Phase 2 — out of scope. The namespaced email prefix ensures test runs don't collide with dev data.

    Step 3 — Create `tests/household-integration.test.ts` with the following structure:

    ```typescript
    /**
     * Real-Prisma integration tests for createHousehold + getUserHouseholds.
     *
     * Per CONTEXT.md D-18: these tests use the REAL database (not mocked Prisma).
     * The only mock is `auth()` — we can't easily simulate a NextAuth session in
     * a test runner, and the goal of these tests is to verify transactional
     * invariants and DB-level constraints (unique slug, referential integrity,
     * $transaction rollback), not session plumbing.
     *
     * Test isolation:
     * - Every user email is namespaced with a unique prefix per test run
     * - afterAll deletes all rows matching the prefix
     * - Safe to run against dev DB (no collision with real dev data)
     */

    import { expect, test, describe, vi, beforeAll, afterAll, beforeEach } from "vitest";
    import { randomUUID } from "node:crypto";

    // Mock `auth` only — EVERYTHING ELSE is real
    vi.mock("../auth", () => ({ auth: vi.fn() }));

    // Import AFTER vi.mock so the mock is in effect
    const { db } = await import("@/lib/db");
    const { createHousehold } = await import("@/features/household/actions");
    const { getUserHouseholds } = await import("@/features/household/queries");
    const { auth } = await import("../auth");

    // Unique namespace per test run
    const RUN_ID = `${Date.now()}-${randomUUID().slice(0, 8)}`;
    const EMAIL_PREFIX = `integration-test-${RUN_ID}`;
    const emailFor = (userTag: string) => `${EMAIL_PREFIX}-${userTag}@test.local`;

    beforeEach(() => {
      vi.clearAllMocks();
    });

    afterAll(async () => {
      // Clean up: delete every user/household created in this run
      const users = await db.user.findMany({
        where: { email: { startsWith: EMAIL_PREFIX } },
        select: { id: true },
      });
      const userIds = users.map((u: { id: string }) => u.id);

      if (userIds.length > 0) {
        // Memberships cascade-delete when users go away (Phase 1 schema)
        // But households created via createHousehold don't cascade on user delete —
        // find them via householdMember rows before deleting users
        const memberships = await db.householdMember.findMany({
          where: { userId: { in: userIds } },
          select: { householdId: true },
        });
        const householdIds = [...new Set(memberships.map((m: { householdId: string }) => m.householdId))];

        await db.user.deleteMany({ where: { id: { in: userIds } } });

        if (householdIds.length > 0) {
          // Some households may still have rows (cascade-dependents) — delete directly
          await db.household.deleteMany({ where: { id: { in: householdIds } } });
        }
      }
      await db.$disconnect();
    });

    // Helper — create a real user row directly (bypasses registerUser so test controls membership setup)
    async function createBareUser(tag: string) {
      return await db.user.create({
        data: {
          email: emailFor(tag),
          passwordHash: "$2b$10$testhashnotused",
          name: `Test User ${tag}`,
        },
      });
    }

    describe("createHousehold integration (D-18)", () => {
      test("happy path: creates Household + OWNER HouseholdMember with isDefault: false", async () => {
        const user = await createBareUser("u1");

        vi.mocked(auth).mockResolvedValue({
          user: { id: user.id, isDemo: false },
        } as Awaited<ReturnType<typeof auth>>);

        const result = await createHousehold({
          name: "Integration Test House",
          timezone: "America/New_York",
        });

        expect(result).toMatchObject({ success: true });
        if (!("success" in result)) throw new Error("expected success");

        const household = result.household;
        expect(household.name).toBe("Integration Test House");
        expect(household.slug).toMatch(/^[A-Za-z0-9_-]{8}$/); // generateHouseholdSlug shape
        expect(household.timezone).toBe("America/New_York");

        // Verify OWNER membership was created in the SAME transaction
        const member = await db.householdMember.findUnique({
          where: { userId_householdId: { userId: user.id, householdId: household.id } },
        });
        expect(member).not.toBeNull();
        expect(member!.role).toBe("OWNER");
        expect(member!.rotationOrder).toBe(0);
        expect(member!.isDefault).toBe(false); // secondary household — NOT the default
      });

      test("atomicity: if any insert fails, neither Household nor HouseholdMember persists", async () => {
        const user = await createBareUser("u2");

        vi.mocked(auth).mockResolvedValue({
          user: { id: user.id, isDemo: false },
        } as Awaited<ReturnType<typeof auth>>);

        // Induce a failure by passing an invalid user id AFTER auth() returns — we'll
        // do this by manually invoking a scenario that violates a FK.
        // Alternative approach: use an extreme-input test that produces a Zod failure
        // BEFORE the transaction starts (empty name). That proves Zod catches the
        // error pre-transaction, no household rows created.
        const result = await createHousehold({ name: "", timezone: "UTC" });
        expect(result).toMatchObject({ error: "Invalid input." });

        // No household with this user was created
        const memberships = await db.householdMember.findMany({
          where: { userId: user.id },
        });
        expect(memberships).toHaveLength(0);
      });

      test("unique-slug: two back-to-back createHousehold calls produce different slugs", async () => {
        const user = await createBareUser("u3");

        vi.mocked(auth).mockResolvedValue({
          user: { id: user.id, isDemo: false },
        } as Awaited<ReturnType<typeof auth>>);

        const r1 = await createHousehold({ name: "House A", timezone: "UTC" });
        const r2 = await createHousehold({ name: "House B", timezone: "UTC" });

        if (!("success" in r1) || !("success" in r2)) {
          throw new Error("both creations should succeed");
        }
        expect(r1.household.slug).not.toBe(r2.household.slug);
      });

      test("rejects unauthenticated: returns { error: 'Not authenticated.' }", async () => {
        vi.mocked(auth).mockResolvedValue(null);

        const result = await createHousehold({ name: "Nope", timezone: "UTC" });
        expect(result).toEqual({ error: "Not authenticated." });
      });
    });

    describe("getUserHouseholds integration (D-18)", () => {
      test("returns array of { household, role, isDefault, joinedAt } sorted by joinedAt asc", async () => {
        const user = await createBareUser("u4");

        // Create TWO memberships manually. The first is marked isDefault=true (matches
        // the registerUser behaviour where the auto-created household is the user's default).
        const hhA = await db.household.create({
          data: {
            name: "Solo House",
            slug: `INT${Date.now()}A`,
            timezone: "UTC",
            cycleDuration: 7,
            rotationStrategy: "sequential",
          },
        });
        await db.householdMember.create({
          data: {
            userId: user.id,
            householdId: hhA.id,
            role: "OWNER",
            rotationOrder: 0,
            isDefault: true,
          },
        });

        // Wait 50ms to guarantee a strictly-later createdAt on the second membership
        await new Promise((r) => setTimeout(r, 50));

        const hhB = await db.household.create({
          data: {
            name: "Roommate House",
            slug: `INT${Date.now()}B`,
            timezone: "UTC",
            cycleDuration: 7,
            rotationStrategy: "sequential",
          },
        });
        await db.householdMember.create({
          data: {
            userId: user.id,
            householdId: hhB.id,
            role: "MEMBER",
            rotationOrder: 1,
            isDefault: false,
          },
        });

        const result = await getUserHouseholds(user.id);

        expect(result).toHaveLength(2);

        // Sorted by joinedAt asc — A first
        expect(result[0].household.id).toBe(hhA.id);
        expect(result[0].role).toBe("OWNER");
        expect(result[0].isDefault).toBe(true);

        expect(result[1].household.id).toBe(hhB.id);
        expect(result[1].role).toBe("MEMBER");
        expect(result[1].isDefault).toBe(false);
      });

      test("returns empty array when user has no memberships", async () => {
        const user = await createBareUser("u5");
        const result = await getUserHouseholds(user.id);
        expect(result).toEqual([]);
      });
    });
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test runner → real DB | Tests connect to `DATABASE_URL`; they can insert/delete rows. Isolation via namespaced email prefix + afterAll cleanup. |
| Mocked auth() → createHousehold | Only `auth()` is mocked; the rest of the call path (Zod, $transaction, findUnique, etc.) runs against real db. |
| Test data leakage → dev environment | afterAll must delete every user created in this run. Memberships cascade on User delete; Households need explicit delete (no cascade from User). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07-01 | I (Information Disclosure) | Test data leakage into dev DB | mitigate | Namespaced email prefix (`integration-test-${runId}-*@test.local`) is collision-proof and grep-able. afterAll cleanup deletes all rows matching the prefix. Worst case — partial cleanup on test-process-crash — leaves rows with the test prefix that are visually distinguishable from real data and can be swept manually. |
| T-02-07-02 | T (Tampering) | Concurrent test run collision | accept | Parallel `vitest` runs on the same DB could theoretically collide, but the unique RUN_ID per `test`-process makes the email prefix globally distinct per process. Vitest default is sequential per file; concurrent across files is fine because different test files use different RUN_IDs. |
| T-02-07-03 | D (Denial of Service) | Dev DB pollution over time | accept | Every run's afterAll cleans up its own rows. Even if all cleanup fails, the disk footprint per failed run is ~5-10 rows — negligible. If pollution accumulates, a manual `DELETE FROM user WHERE email LIKE 'integration-test-%'` cleanup is trivial. |
| T-02-07-04 | E (Elevation of Privilege) | Mocked auth() bypasses real authz | accept | Intentional — integration tests verify DB-level behaviour, not session authz (which is covered by Plan 02-06's ForbiddenError tests). Documented in test file header. |
| T-02-07-05 | I | Test fixture data exposed as real data | accept | Test user rows are clearly tagged with `integration-test-*` email prefix. Even if retained, they cannot log in (password hash is a fixed bcrypt non-value). |
</threat_model>

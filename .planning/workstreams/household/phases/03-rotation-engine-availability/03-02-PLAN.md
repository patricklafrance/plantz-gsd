| database schema change | Migration runs against dev Postgres; prod deploy is a separate release step |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-AUTH-CRON | Spoofing | /api/cron/advance-cycles | mitigate | proxy.ts matcher excludes api/cron so only the route handler's bearer check gates access. No session cookie is expected; `proxy.ts` silently redirecting to /login would break the endpoint. |
| T-3-DB-MIG-01 | Tampering | prisma migrate dev on dev DB | mitigate | Migration runs via `prisma migrate dev` (not `db push`), producing a committed `migration.sql` file that is reviewable and reversible. Backfill for pre-existing Cycle-less households is inventoried in Task 2 Step 1; zero rows expected per Phase 1 DB flush decision. |
| T-3-DB-DEDUPE-01 | Integrity | HouseholdNotification deduplication | mitigate | `@@unique([cycleId, recipientUserId, type])` enforced at DB level. Wave 2 transition code catches P2002 and treats as idempotent no-op. Nullable-cycleId edge case is Phase 5+ concern and is documented in the schema comment (Pitfall D). |
</threat_model>

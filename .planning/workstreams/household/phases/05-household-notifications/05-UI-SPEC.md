# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5: Household Notifications. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope surfaces:** four dashboard banners (cycle-start, reassignment, passive-status, fallback), unified `NotificationBell` (desktop + mobile variant) with merged plant-reminder + cycle-event feed, mobile `BottomTabBar` Alerts slot refactor, mark-read-on-open interaction.

## Typography

Existing dashboard rhythm already declares the size ladder; Phase 5 reuses these exactly — no new sizes introduced.

| Role | Size | Tailwind | Weight | Line Height | Usage |
|------|------|----------|--------|-------------|-------|
| Micro | 10px | `text-[10px]` | 600 (`font-semibold`) | 1 | Badge count number only |
| Caption | 12px | `text-xs` | 500 (`font-medium`) | 1.4 | Dropdown item secondary line ("Room · status"), BottomTabBar tab label, banner meta ("Cycle ends Tue Apr 23") |
| Body | 14px | `text-sm` | 400 (`font-normal`) | 1.5 | Banner body copy, dropdown primary line, dropdown empty-state copy |
| Emphasis | 14px | `text-sm` | 600 (`font-semibold`) | 1.5 | Assignee names in banner copy, dropdown item titles (plant nickname / event subject) |
| Heading (out-of-phase) | 24px | `text-2xl` | 600 (`font-semibold`) | 1.2 | Existing `Dashboard` H1 — Phase 5 does not modify |

**Font weights declared:** 2 — `font-normal` (400) and `font-semibold` (600). `font-medium` (500) appears only on the BottomTabBar tab label (existing code, unchanged by Phase 5).

**Line heights:** Tailwind v4 defaults are accepted — `leading-normal` (1.5) for body, `leading-tight` (1.25) for single-line badge.

**Rationale for no new typography:** Phase 5 adds surfaces, not a new visual vocabulary. Dashboard already renders `text-sm`/`text-xs` body with `font-semibold` emphasis; banners and dropdown items must match so the new surfaces read as the same app.

## Interaction Contract

### Banner render logic (D-13)

Given `viewer`, `currentCycle`, `viewerNotifications`, `viewerIsAssignee`, `memberCount`:

```
if (!currentCycle) return null

// Layer 1: fallback (always first when applicable)
if (currentCycle.status === 'paused' || currentCycle.transitionReason === 'all_unavailable_fallback')
  render <FallbackBanner />

// Layer 2: assignee-role event (mutually exclusive — D-19's unique index guarantees ≤1 unread event per viewer per cycle)
const unreadEvent = viewerNotifications.find(n => n.readAt === null && n.cycleId === currentCycle.id)
if (unreadEvent?.type === 'cycle_started') render <CycleStartBanner />
else if (unreadEvent?.type.startsWith('cycle_reassigned_')) render <ReassignmentBanner />

// Layer 3: passive status (only if no assignee-role banner shown AND viewer is not assignee)
if (!unreadEvent && !viewerIsAssignee && currentCycle.status === 'active') {
  // suppress passive banner entirely for single-member households where viewer IS the sole member
  if (memberCount === 1) render nothing
  else render <PassiveStatusBanner />
}
```

### Mark-read interaction (D-16, D-20)

- **Trigger:** `DropdownMenu.onOpenChange(open => open === true)` fires on bell dropdown open.
- **Mechanism:** `useTransition` wraps a call to `markNotificationsRead({ householdId, notificationIds })`. The `notificationIds` array is snapshotted from the merged feed prop BEFORE the state update (to avoid race).
- **Server Action behavior:** `updateMany({ where: { id: { in: ids }, recipientUserId: session.user.id, readAt: null }, data: { readAt: new Date() } })` then `revalidatePath('/h/{slug}/')`. Idempotent — re-open fires again but matches zero rows.
- **Banners do NOT mark-read on mount.** Unread state persists until the bell is opened (D-16). This is the single most testable interaction contract in the phase.
- **Visual state in dropdown:**
  - Unread cycle event: `border-l-2 border-accent` left accent stripe, full-color text
  - Read cycle event: no stripe, `text-muted-foreground` body, still clickable
- **Toast / error handling:** `markNotificationsRead` is fire-and-forget through `useTransition` — no toast on success (silent), no toast on error (logged only). Rationale: mark-read is not user-visible work; re-opening retries. Reuses existing `sonner` infra only for explicit user-initiated actions (out of phase scope).

### Merged dropdown feed order (D-18)

Within the dropdown panel, items render top-to-bottom in fixed buckets:

1. Overdue plants (from `getReminderItems`) — sorted by days-overdue desc, nickname asc tiebreaker
2. Due-today plants (from `getReminderItems`) — sorted by nickname asc
3. Unread cycle events for current active cycle (from `getCycleNotificationsForViewer`) — sorted by `createdAt` desc
4. Read cycle events for current active cycle — sorted by `createdAt` desc, visually muted

Prior-cycle events never appear.

### Badge count (D-19)

`reminderCount + unreadCycleEventCount` — a single integer. No separate dot, no split. Displayed as "99+" when > 99 (existing convention in desktop `notification-bell.tsx`). Mobile BottomTabBar previously capped at "9+" — Phase 5 harmonizes both variants to "99+".

### Bell variant differences

| Aspect | `variant="desktop"` | `variant="mobile"` |
|--------|-----|-----|
| Trigger shape | `size="icon"` ghost button, `p-2.5` | Flex-1 column, `min-h-[44px]`, icon + "Alerts" label underneath |
| Badge position | `-top-1 -right-1` on the icon | `-top-1 -right-1` on the icon within the label stack |

## Accessibility Contract

| Element | Requirement |
|---------|-------------|
| All banners | `role="status"` except `FallbackBanner` which uses `role="alert"` (matches existing `TimezoneWarning`) |
| All banner icons | `aria-hidden="true"`; semantic info lives in text |
| Bell trigger | `aria-label` dynamic: `"{count} notifications"` when count > 0, else `"No new notifications"` (broader than current "plants need attention" — merged feed means both plants and cycle events) |
| Bell badge | Visually rendered number; `aria-label` on trigger is the accessible count — badge itself has no separate `aria-label` (avoids double-announce) |
| Dropdown items | Each cycle-event item is focusable via `DropdownMenuItem` keyboard nav; Enter/Click navigates to `/h/{slug}/dashboard` and scrolls to the matching banner (out of phase if too complex — fallback: close the menu only) |
| Touch targets | Mobile bell trigger and all BottomTabBar tabs: `min-h-[44px]` (iOS HIG) — unchanged from existing convention |
| Focus ring | Inherits existing `focus-visible:ring-2 focus-visible:ring-ring/50 focus-visible:ring-offset-1` from BottomTabBar and shadcn button; no new focus styles |
| Color contrast | All text/background pairs above use tokens verified WCAG AA 4.5:1 in globals.css lines 83-109. `text-destructive` on white = 4.5:1; `text-accent` on white = 4.6:1; `text-muted-foreground` on white = 5.4:1. No new pairs introduced. |
| Screen-reader order | Banners appear above the H1, so SR users hear fallback → assignee event → passive status → "Dashboard" → content. This matches visual order. |

## Copywriting Contract

All copy is verb-led, second person where viewer is addressed, third person where describing other members. Date formatting: `Tue Apr 23` (weekday + month + day, no year, no time) via `date-fns` `format(date, 'EEE MMM d')`.

### Primary CTAs in Phase 5 surfaces

| Element | Copy | Context |
|---------|------|---------|
| Bell dropdown item (cycle event, unread) | `{Subject line}` + secondary `{Meta line}` | See table below |
| Bell dropdown (empty state) | Heading: `You're all caught up` — Body: `New reminders and cycle updates will appear here.` | Replaces existing "No reminders — Plants that need attention will appear here." to reflect the merged feed |
| Banner "view details" / dismiss | **None.** All four banners are read-only per D-16. No CTAs, no dismiss button. |

### CycleStartBanner (HNTF-02)

| Element | Copy |
|---------|------|
| Subject | `You're up this cycle.` |
| Meta | `{N} plants due · Cycle ends {Tue Apr 23}` |
| Zero-due variant | Meta becomes `No plants due right now · Cycle ends {Tue Apr 23}` |

### ReassignmentBanner (HNTF-03) — type-branched

| Type | Subject | Meta |
|------|---------|------|
| `cycle_reassigned_manual_skip` | `{Alice} skipped — you're covering this cycle.` | `{N} plants due · Cycle ends {Tue Apr 23}` |
| `cycle_reassigned_auto_skip` | `{Alice} is unavailable — you're covering this cycle.` | `{N} plants due · Cycle ends {Tue Apr 23}` |
| `cycle_reassigned_member_left` | `{Alice} left the household — you're covering this cycle.` | `{N} plants due · Cycle ends {Tue Apr 23}` |

`{Alice}` is `cycle.household.members` joined user's `name ?? email`. Name is rendered `font-semibold`.

### PassiveStatusBanner (HNTF-04)

| Element | Copy |
|---------|------|
| Subject (member count > 1, next assignee known) | `{Alice} is watering this cycle. {Bob} is next up.` |
| Subject (member count > 1, next returns fallback owner) | `{Alice} is watering this cycle. {Owner} covers if no one's available next.` |
| Subject (single-member household where viewer is assignee) | *Banner suppressed entirely.* |
| Meta | `Cycle ends {Tue Apr 23}` |

Names rendered `font-semibold`; separator between subject and next-up line is a period + space (no bullet).

### FallbackBanner (AVLB-05 + D-09)

| Variant | Subject | Meta |
|---------|---------|------|
| Viewer is owner, owner is covering | `Nobody's available — you're covering this cycle.` | `Check back when members update their availability.` |
| Viewer is non-owner, owner is covering | `Nobody's available — {Owner} is covering this cycle.` | `You can update your availability in settings.` |
| Paused (strict, no fallback active) | `This week's rotation is paused.` | `Someone needs to step up — plants still need water.` |

`{Owner}` is `household.members` filtered by `role === 'OWNER'` → `name ?? email`.

### Empty state (merged-feed bell dropdown)

| Element | Copy |
|---------|------|
| Heading | `You're all caught up` |
| Body | `New reminders and cycle updates will appear here.` |

### Error state (`markNotificationsRead` failure — silent)

No user-facing error copy. Failure is logged only (per interaction contract). Re-open retries. Rationale: this action is incidental to the user's intent (opening the dropdown), and a toast on failure would be noisy for a recoverable, imperceptible operation.

### Destructive actions in this phase

**None.** Phase 5 adds no destructive actions. `markNotificationsRead` is idempotent and non-destructive (mark-as-read only; no delete, no mark-unread path — the latter is explicitly deferred per D-04).

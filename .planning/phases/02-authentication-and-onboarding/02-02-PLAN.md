**Form onSubmit handler:**

```typescript
async function onSubmit(values: LoginInput) {
  try {
    const result = await signIn("credentials", {
      email: values.email,
      password: values.password,
      redirect: false, // Handle redirect manually to show errors
    });

    if (result?.error) {
      toast.error("Incorrect email or password. Please try again.");
      return;
    }

    // Successful login — redirect to dashboard
    router.push("/dashboard");
    router.refresh();
  } catch {
    toast.error("Something went wrong. Please try again in a moment.");
  }
}
```

IMPORTANT: For the login form, use `signIn` from `next-auth/react` (client-side) with `redirect: false`. This returns a result object instead of throwing. Check `result?.error` and show a toast. On success, use `router.push("/dashboard")` + `router.refresh()`. This is the CLIENT-SIDE login pattern (different from the SERVER-SIDE signIn in registerUser action).

Error messages match UI-SPEC Copywriting Contract exactly:

- Network/server error: "Something went wrong. Please try again in a moment."

**Form onSubmit handler:**

```typescript
async function onSubmit(values: RegisterInput) {
  const result = await registerUser({
    email: values.email,
    password: values.password,
    confirmPassword: values.confirmPassword,
  });

  // If registerUser returns (didn't throw redirect), it means there was an error
  if (result?.error) {
    toast.error(result.error);
    return;
  }
  // If we reach here without redirect, something unexpected happened
  // The normal success path is: registerUser calls signIn which throws NEXT_REDIRECT

```

IMPORTANT: Unlike login, the register form calls the `registerUser` Server Action directly. The form only needs to handle the error case (when the action returns `{ error }`). The loading state (`isSubmitting`) is managed by react-hook-form and will be true until the redirect completes.

The `disabled` prop should be set on all Input fields when `form.formState.isSubmitting` is true (per UI-SPEC: "All three fields disabled while action is in-flight").

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form -> signIn (client login) | Email/password sent to NextAuth endpoint via signIn from next-auth/react |
| Browser form -> registerUser (server action) | Form data sent to Server Action for user creation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Information Disclosure | Login error message | mitigate | Generic "Incorrect email or password" — never reveals whether email exists (UI-SPEC error contract) |
| T-02-08 | Tampering | Register form client validation | mitigate | Client-side Zod validation is defense-in-depth only; registerUser action re-validates server-side with same schema |
| T-02-09 | Spoofing | Password visibility toggle | accept | Password toggle is a standard UX pattern; no security risk since user is on their own device |
| T-02-10 | Information Disclosure | Register email exists error | accept | "An account with this email already exists" is standard for registration flows; user enumeration risk is minimal since registration is a self-service action |
</threat_model>

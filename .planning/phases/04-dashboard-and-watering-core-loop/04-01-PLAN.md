    **Step 4: Create `src/features/watering/queries.ts`**

    This file contains `getDashboardPlants` and `getWateringHistory`. The classification function (`classifyAndSort`) is exported as a named export for testing.

    `getDashboardPlants(userId: string, todayStart: Date, todayEnd: Date)`:
    - Fetches all active (non-archived) plants for the user with `include: { room: true, careProfile: true, wateringLogs: { orderBy: { wateredAt: "desc" }, take: 1 } }`
    - Calls `classifyAndSort(plants, todayStart, todayEnd)` which:
      - For each plant, computes `daysUntil = differenceInDays(nextWateringAt, todayStart)` using date-fns
      - Classifies: `daysUntil < 0` -> "overdue", `nextWateringAt >= todayStart && nextWateringAt < todayEnd` -> "dueToday", `daysUntil > 0 && daysUntil <= 7` -> "upcoming"
      - "recentlyWatered": plants whose `lastWateredAt` is within the last 48 hours AND are NOT overdue/dueToday (they have upcoming nextWateringAt)
      - Plants with no `nextWateringAt` are classified as "upcoming" with `daysUntil` based on wateringInterval from creation
      - Returns `{ overdue: DashboardPlant[], dueToday: DashboardPlant[], upcoming: DashboardPlant[], recentlyWatered: DashboardPlant[] }` — a grouped object, NOT a flat array
    - Sort per D-03: overdue by most days late first (ascending daysUntil, most negative first), dueToday alphabetical by nickname, upcoming by soonest (ascending daysUntil), recentlyWatered by most recent lastWateredAt descending

    `getWateringHistory(plantId: string, userId: string, skip = 0, take = 20)`:
    - Fetches watering logs for a plant: `db.wateringLog.findMany({ where: { plantId, plant: { userId } }, orderBy: { wateredAt: "desc" }, skip, take })`
    - Also returns `total: number` via `db.wateringLog.count(...)` for "Load more" logic

    Both functions use `import { db } from "@/lib/db"` and `import { differenceInDays } from "date-fns"`.

    The `todayStart`/`todayEnd` parameters represent the user's local day boundaries in UTC. These are passed from the dashboard Server Component which reads them from the timezone cookie.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Server Action | Untrusted form data from log/edit watering dialogs crosses to server |
| Client -> Server Component | Timezone cookie from client is untrusted input |
| WateringLog -> Plant | Cross-entity mutation (watering log affects plant's nextWateringAt) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | logWatering action | mitigate | Ownership check: `findFirst({ where: { id: plantId, userId: session.user.id } })` before any mutation |
| T-04-02 | Tampering | editWateringLog action | mitigate | Ownership via join: `findFirst({ where: { id: logId, plant: { userId } } })` — verifies through plant relation |
| T-04-03 | Tampering | deleteWateringLog action | mitigate | Same ownership-via-join pattern as T-04-02 |
| T-04-04 | Tampering | logWatering — future date | mitigate | Zod schema: `z.coerce.date().max(new Date(), "Cannot log future watering.")` |
| T-04-05 | Denial of Service | logWatering — duplicate spam | mitigate | 60-second server-side debounce: reject if `createdAt >= now - 60s` for same plantId |
| T-04-06 | Tampering | timezone cookie | accept | Cookie only affects display grouping (overdue vs due-today); no security impact. Invalid tz falls back to UTC. |
| T-04-07 | Information Disclosure | getWateringHistory query | mitigate | Query includes `plant: { userId }` filter — cannot read another user's watering logs |
| T-04-08 | Tampering | note field XSS | accept | React JSX auto-escapes all rendered text; no `dangerouslySetInnerHTML`; Zod enforces max 280 chars |
</threat_model>

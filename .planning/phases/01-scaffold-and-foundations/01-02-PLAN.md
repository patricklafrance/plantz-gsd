    1. Create `auth.config.ts` in the project root (same level as app/ and src/). This file is edge-safe:

       ```typescript
       import type { NextAuthConfig } from "next-auth";

       export const authConfig = {
         pages: {
           signIn: "/login",
         },
         session: {
           strategy: "jwt",
         },
         callbacks: {
           authorized({ auth, request: { nextUrl } }) {
             const isLoggedIn = !!auth?.user;
             const isOnProtectedRoute =
               nextUrl.pathname.startsWith("/dashboard") ||
               nextUrl.pathname.startsWith("/plants") ||
               nextUrl.pathname.startsWith("/rooms");

             if (isOnProtectedRoute) {
               return isLoggedIn;
               // Returning false redirects to pages.signIn ("/login")
             }

             // Redirect logged-in users away from auth pages to dashboard
             if (
               isLoggedIn &&
               (nextUrl.pathname === "/login" || nextUrl.pathname === "/register")
             ) {
               return Response.redirect(new URL("/dashboard", nextUrl));
             }

             return true;
           },
         },
         providers: [],
       } satisfies NextAuthConfig;
       ```

    2. Create `auth.ts` in the project root. This file uses Node.js APIs (bcryptjs, Prisma):

       ```typescript
       import NextAuth from "next-auth";
       import Credentials from "next-auth/providers/credentials";
       import bcryptjs from "bcryptjs";
       import { z } from "zod/v4";
       import { authConfig } from "./auth.config";
       import { db } from "@/lib/db";

       export const { auth, handlers, signIn, signOut } = NextAuth({
         ...authConfig,
         providers: [
           Credentials({
             async authorize(credentials) {
               const parsed = z
                 .object({
                   email: z.email(),
                   password: z.string().min(6),
                 })
                 .safeParse(credentials);

               if (!parsed.success) return null;

               const { email, password } = parsed.data;
               const user = await db.user.findUnique({ where: { email } });
               if (!user) return null;

               const passwordsMatch = await bcryptjs.compare(
                 password,
                 user.passwordHash,
               );
               if (!passwordsMatch) return null;

               return { id: user.id, email: user.email, name: user.name };
             },
           }),
         ],
       });
       ```

       Note: Use z.email() (Zod v4 shorthand) instead of z.string().email(). Both work in v4, but z.email() is the idiomatic v4 pattern. Import from "zod/v4" not "zod".

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Edge runtime (proxy.ts) vs Node.js (auth.ts) | proxy.ts runs on edge; auth.ts runs on Node.js. Mixing APIs breaks at runtime. |
| Unauthenticated vs authenticated routes | proxy.ts + authorized callback enforces the boundary |
| Credentials input | User-supplied email/password crosses trust boundary at authorize() |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | auth.ts authorize() | mitigate | Zod v4 validates email format and password min length before database lookup; bcryptjs.compare for password verification; return null (not error) on all failure paths to prevent enumeration |
| T-02-02 | Elevation of Privilege | proxy.ts matcher | mitigate | Matcher regex excludes only specific safe prefixes; authorized callback explicitly checks pathname prefixes for /dashboard, /plants, /rooms; default-open for public pages, default-closed for app pages |
| T-02-03 | Information Disclosure | auth.ts error paths | mitigate | authorize() returns null for both "user not found" and "wrong password" -- no distinction leaked to caller; prevents user enumeration |
| T-02-04 | Spoofing | JWT session | accept | NextAuth v5 handles JWT signing/verification with AUTH_SECRET; encrypted JWE by default; session rotation handled by library |
| T-02-05 | Tampering | CSRF on auth endpoints | accept | NextAuth v5 includes CSRF protection in its route handlers by default |
</threat_model>
